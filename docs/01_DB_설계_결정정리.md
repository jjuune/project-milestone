## 🧩 1. 데이터를 담는 5개의 "통" (테이블)

PITSTOP이 다루는 데이터를 5개의 통으로 나눴어.

| 통 이름 | 담는 내용 | 쉬운 비유 |
| :--- | :--- | :--- |
| **organizations** | 회사 정보 | "우리 회사" 자체 |
| **profiles** | 직원 정보 | 직원 명함 |
| **vehicles** | 차량 기본 정보 | 차량 명패 (이름, 차종, 연료, 현재 거리) |
| **vehicle_wallets** | 차량의 "고정 정보" | 차량 서류 보관함 (보험사, 검사일, 차량번호 등 — 자주 안 바뀌는 것들) |
| **vehicle_events** | 차량에 일어난 모든 일 | 차량 일기장 (오일 교체, 거리 입력, 보험 갱신 등 시간순 기록) |

**왜 wallet과 events를 따로 나눴나?**
- Wallet = "거의 안 바뀌는 정보" (보험사, 차량번호)
- Events = "계속 쌓이는 기록" (오늘 오일 갈았음, 어제 거리 입력했음)

이 둘을 같은 통에 넣으면 나중에 헷갈리고 관리하기 어려워져서, 처음부터 분리해뒀어.

## 🔑 2. 누가 뭘 볼 수 있는가 (권한)

논의 끝에 정리된 결론은 이거야:

> **같은 회사 직원이면 누구나 모든 차량을 볼 수 있고, 수정도 할 수 있다.**
> 단, "누가 그 기록을 입력했는지"는 자동으로 남는다.

**왜 이렇게 정했나?**
- 처음엔 "이 직원은 이 차만, 저 직원은 저 차만" 같은 세분화된 권한을 고민했는데, 실제로는 직원들이 여러 차를 교차로 쓰는 상황이라 그렇게 나누면 오히려 불편함만 커짐
- 회사 차량을 운전할 수 있다는 것 자체가 이미 회사의 신뢰를 받은 상태라서, 굳이 "이 사람은 못 보게" 막을 이유가 없음
- 외부 사람(정비소, 보험사 직원 등)은 시스템에 직접 들어오지 않고, 그 일을 처리한 **내부 직원이 대신 기록**하는 방식으로 처리

**그래도 남겨두는 안전장치**
- 모든 기록에는 "누가 입력했는지"가 자동으로 같이 저장됨
- 이건 누군가를 의심하려는 게 아니라, 나중에 "어? 이 기록 좀 이상한데?" 싶을 때 누구한테 물어보면 되는지 알기 위한 최소한의 장치

## 📋 3. 차량에 기록되는 "이벤트"들

엔진오일 교체든, 단순히 거리만 입력하는 것이든, **전부 하나의 "이벤트"로 같은 곳에 기록**돼.

```
2026.06.24   엔진오일 교체   50,000km   [기록: 김대리]
2026.06.10   거리 입력만    49,200km   [기록: 박과장]
2026.03.11   와이퍼 교체    47,300km   [기록: 김대리]
```

**왜 "거리 입력"도 똑같은 이벤트로 처리하나?**
- 둘을 따로 나누면 나중에 "전체 차량 히스토리를 시간순으로 보여줘" 할 때 두 군데를 다 뒤져서 합쳐야 해서 번거로움
- 하나로 합쳐두면 타임라인 보여줄 때 그냥 한 줄로 쭉 나옴
- 나중에 혹시 "운행일지" 같은 기능이 필요해지면, 지금 구조를 바꾸지 않고 "거리 입력 기록만 따로 뽑아 보는 창"을 하나 추가하면 돼 (테이블을 새로 만들 필요 없음)

이벤트 종류는 자유롭게 추가될 수 있게(기타 항목이 늘어날 가능성이 높다고 판단해서) 딱딱하게 고정된 목록이 아니라 **유연하게 새 항목을 넣을 수 있는 방식**으로 설계했어.

## ⚡ 4. "현재 누적거리"는 어떻게 빠르게 보여주나

대시보드에서 차량 카드 넘길 때마다 "이 차 지금까지 몇 km 탔지?"를 매번 처음부터 계산하면 느려져. 그래서 `vehicles` 통에 "현재 누적거리"를 따로 한 번 더 저장해두고, **새 이벤트가 들어올 때마다 자동으로 갱신**되게 했어.

**여러 사람이 동시에 입력해도 괜찮은가?**
- 갱신하는 "주체"를 딱 하나로 정해뒀어 (데이터베이스 자체가 처리) — 여러 직원이 동시에 입력해도 충돌 없이 순서대로 안전하게 처리됨
- 혹시 누가 며칠 전 날짜로 늦게 입력해도, 항상 "실제로 가장 최근 날짜"의 기록이 기준이 되도록 처리해서 꼬일 일이 없음
