# 🚗 Vehicle Log MVP 기획서 v2.0

## 1. 프로젝트 개요
* **서비스명**: PITSTOP (Code name : Milestone)
* **한 줄 정의**: 법인 차량에 대한 정보를 기억할 필요 없도록 만들어주는 차량 관리 대시보드
* **목표**: 차량 관리를 위한 복잡한 ERP를 만드는 것이 아니라, 차량과 관련된 정보와 이력을 한 곳에 저장하고 필요한 시점에 쉽게 찾을 수 있도록 한다.

---

## 2. 핵심 문제 정의
> **"소규모 사업장(1~5대 차량)의 차량 관리는 대부분 담당자의 기억에 의존한다."**

* **대표적인 사례 (의문점들)**:
  * *엔진오일 언제 교체했지?*
  * *보험사가 어디였지?*
  * *정기검사 언제였지?*
  * *이 차 타이어 규격이 뭐였지?*
  * *지난번 정비는 언제 했지?*

이러한 정보는 자주 사용하지 않지만 필요할 때 매우 중요하다.

---

## 3. 제품 철학

### 🚫 하지 않는 것 (Don't)
* 배차관리
* 운전자관리
* 운행일지 자동생성
* 비용분석 ERP
* 복잡한 정비 관리

### 🟢 하는 것 (Do)
* 차량 상태를 한눈에 보여준다.
* 차량 관련 기억을 저장한다.
* 필요한 시점에 알림을 제공한다.
* 차량 이력을 쉽게 남긴다.

* **핵심 엔티티**: `Vehicle` 보다 `Vehicle Event` 중심

### 🛠️ Event 종류 (Event 종류)

* 타이어 교체
* 브레이크 교체
* 배터리 교체
* 와이퍼 교체
* 보험 갱신
* 정기검사 완료

### 📥 입력 UX
* **목표**: `10초 이내 기록 완료`
* **사용자 흐름 (사용자 흐름)**:
  1. `[+]` 버튼 클릭
  2. **무엇을 했나요?**:
     * `○ 엔진오일` / `○ 타이어` / `○ 브레이크` / `○ 배터리` / `○ 와이퍼` / `○ 기타`
  3. **현재 주행거리 입력**:
     * `50,000km`
  4. **영수증 사진 첨부**
  5. **저장**
* **생성 결과**:
  * `2026.06.24` / `엔진오일 교체` / `50,000km` / `영수증 사진`

## 📡 10. 진행 중인 검증 및 개선 방향 (NFC & UX)

### ① 입력 트리거 고도화: NFC 태깅 연동
* **기획 방향**: 차량 탑승 시 빠른 입력을 위해 실물 NFC 카드(NTAG215)를 활용한 1차 입력 검증 진행.
* **하이브리드 대응**: 스캔 디바이스 부재 및 Fail-safe 대응을 위해 **[NFC + QR 병행 안내 라벨]**을 차량 내 콘솔 트레이 등 동작 마찰이 가장 적은 곳에 거치하는 방식으로 실물 거치 기획.
* **NTAG215 락(Lock) 이슈 예외 처리**:
  * 카드 락 해제 상태일 시: NDEF URI 레코드 기록을 통해 태깅 즉시 디폴트 입력 창 실행 유도.
  * 카드 락 걸린 상태일 시: 카드의 고유 하드웨어 UID를 읽어 Supabase DB 내 차량 식별 키값과 매핑하여 분기시키는 대체 아키텍처 수립.

### ② 입력 인터페이스 분석: 텔레그램 봇 vs PWA Standalone 앱 (검토 진행 중)
* **텔레그램 챗봇 형태**:
  * *장점*: 텔레그램 고유 세션 ID(`chat.id`)를 기반으로 운전자를 완벽하게 자동 인증할 수 있어 명의 도용 리스크가 차단됨.
  * *과제*: 오타가 발생했을 때 DB를 오염시킬 리스크가 있어, 입력 단계에서 **"직전 주행거리 대비 증가량 표시 및 재확인 컨펌"** 단계를 의무 탑재하여 정합성 보완 필요.
* **PWA (Progressive Web App) Standalone 패키징 형태**:
  * *장점*: 모바일 홈 화면에 앱 아이콘을 추가하여 standalone 전체 화면으로 구동. 오프라인 캐싱(IndexedDB 및 Service Worker) 기능을 이용해 지하 주차장 등 통신 음영 지역에서도 이벤트 입력이 끊기지 않는 오프라인 퍼스트(Offline-First) 및 백그라운드 동기화 지원. 네이티브 카메라 직접 링크 가능.
  * *과제*: iOS 기기의 경우 웹 푸시 수신을 위해 수동 홈 화면 추가 유도 가이드라인 등 초기 진입 튜토리얼 설계 필요.
* **결론**: 두 방식의 실제 스캔 및 입력 입력 효율성을 직접 필드 테스트한 후 최종 스펙 결정 예정.

### ③ 차량 운행 일지용 기초 데이터(Raw Data) 활용
* 국세청 기준 세무 리스크가 있는 패턴 기반 '가상 운행 자동 쪼개기(Epic 4)'는 기획에서 공식 보류함.
* 단, 시스템 내에 실제로 입력되는 소모품 정비 및 주행거리 이벤트 이력 데이터를 활용하여, **사후 수동 운행 일지 작성 시 신뢰할 수 있는 로 데이터(Raw Data) 및 주행 마일스톤 증빙 근거로 요긴하게 연동·재활용**할 수 있는 관리자 대시보드 뷰어는 MVP 스펙으로 유지함.
