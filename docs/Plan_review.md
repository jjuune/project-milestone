🔍 1. 전체 설계 검토 — 크리티컬 이슈 & 과설계 신호
크리티컬 문제

주행거리 입력 UX: 텔레그램에서 사용자가 숫자를 직접 타이핑하는 방식은 오타로 인한 DB 오염 위험이 큼 → "직전 대비 변화량 표시 + 확인 버튼" 컨펌 단계 추가 권장
Epic 4 (운행일지 자동 쪼개기): 주 1회 입력으로 나머지 요일 운행 기록을 패턴 기반으로 "가상 생성"하는 건 국세청 운행기록부의 취지(실제 운행 사실 입증)와 충돌할 수 있어 법적 리스크 있음

과설계 신호

3색 네온 Glassmorphism 다크테마 → 렌더링 비용 대비 체감 효용 낮음, 반응 속도가 더 중요
파워트레인별 소모품 "가변 렌더링 엔진" → fuel_type 컬럼 하나 + 프론트 필터링으로 충분
SHA-256 해시 비교 자체 구현 어드민 인증 → Supabase Auth 활용이 더 안전하고 코드도 적음


⏸️ 2. Epic 4 — 보류 확정

1단계 작업 범위에서 제외
세무사 확인 후 재검토하는 것으로 결정
백로그 문서에 보류 이유(법적 리스크 검토 필요)를 주석으로 남겨두는 걸 추천했었음 — 아직 문서 자체는 수정 안 한 상태니까, 필요하면 반영해줄게


📡 3. NFC 트리거 방식 — 검증 진행 중
결정된 방향

QR 대신 NFC 태그를 1차 검증 대상으로 선택
보유 중인 신용카드 사이즈 빈 NFC 카드(아미보 카피용 NTAG215) 활용
NFC + QR 병행 안내 라벨도 함께 고려 중 (fallback 경로 확보 목적)

부착 위치 — 차량 표면 직접 부착에서 "거는 방식"으로 전환

대시보드 직접 부착 / 룸미러 매달기 / 햇빛가리개 / 콘솔 트레이 비교
룸미러 매달기는 시야 방해로 단속 대상 가능성 있어 주의 필요
콘솔 트레이에 두는 방식이 동작 마찰이 가장 적은 대안으로 제안됨 (최종 확정은 아직 안 함)

기술적 확인 필요 사항

NTAG215 칩은 카피 과정에서 Lock Bit가 같이 구워져 재기록이 안 되는 경우가 있음 (jjuune이 기억하는 "한 번 쓰면 수정 불가" 현상의 원인으로 추정)
내일 NFC Tools 앱으로 Lock 상태 / Memory 정보 먼저 스캔 → 쓰기 가능 여부 확인 후 진행
만약 락 걸려있으면: 다시 쓰기 대신 UID 자체를 차량 식별자로 활용하고 매핑은 Supabase DB에서 관리하는 대안도 있음
다행히 빈 카드 재고가 충분해서 락 걸린 카드가 나와도 다른 카드로 바로 시도 가능
