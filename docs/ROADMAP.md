# PITSTOP V1 (MVP) 개발 로드맵

> 확장성을 최우선 기준으로 설계한 V1 실행 계획
> 범위: MVP(V1)까지 상세화, V1.5/V2는 "막지 않는 선택"만 미리 고려

---

## 🏗️ 기술 스택 결정

| 영역 | 선택 | 역할 |
| :--- | :--- | :--- |
| **Backend / DB** | Supabase | 데이터, Auth, Storage, Edge Functions(알림) — 항상 켜져 있어야 하는 코어 |
| **Frontend** | Vite + React (PWA) | 대시보드 UI, 홈 화면 추가로 앱처럼 사용 |
| **NAS (Synology)** | 백업 전용 | Supabase DB 정기 덤프 저장 — 클라우드 장애 + 백업 부재의 이중 리스크 방지 |
| **n8n** | 부가 자동화 (선택) | 텔레그램 알림 중계 등 "있으면 좋은" 채널, 핵심 기능과 분리 |

**설계 원칙**: NAS/n8n 장애가 PITSTOP 핵심 기능(조회·입력·알림)에 영향을 주지 않도록 분리. NAS는 메인 의존이 아니라 보강 역할로 한정.

## 1단계. DB 스키마 + Auth 설계 ⭐ (가장 중요)

> 여기서 잘못되면 다중 사용자 권한 구조를 나중에 갈아엎어야 함 — 가장 신중하게 짤 단계

### 핵심 테이블 (초안)

- `vehicles` — 차량 기본 식별 정보
- `vehicle_wallets` — 보험사, 만기일, 긴급출동 번호 등 정적 정보 (vehicles와 1:1이지만 분리 보관)
- `vehicle_events` — 이벤트 중심 설계의 핵심 (교체/갱신/검사 등), `recorded_by` 필드로 다중 사용자 기록자 추적
- `vehicle_members` — 사용자-차량 권한 매핑 (다중 사용자 대응)
- Auth — Supabase Auth 기본 제공 (이메일/비밀번호 또는 매직링크)

### 해야 할 일

- [ ] 테이블 스키마 확정 (컬럼, 타입, 관계)
- [ ] RLS(Row Level Security) 정책 설계 — "소속된 차량만 조회/수정 가능"
- [ ] 소모품 주기 기준값 테이블 분리 여부 결정 (하드코딩 vs 설정 테이블)

## 🔭 V1.5 / V2 확장을 "막지 않기 위해" V1에서 챙길 것

| 항목 | V1에서 할 일 | 이유 |
| :--- | :--- | :--- |
| 사진 첨부 (V1.5) | `vehicle_events`에 `receipt_url` 컬럼 미리 추가 (NULL 허용) | 나중에 컬럼 추가 마이그레이션 불필요 |
| OCR/AI 보조 (V2) | Edge Function 구조로 알림 로직 작성 | 동일 구조에 AI 호출 함수만 추가하면 확장 가능 |
| 다중 사용자 | `recorded_by`, `vehicle_members` 테이블 V1부터 반영 | 나중에 권한 구조 추가가 아니라 처음부터 내장 |
| NAS 백업 | DB 덤프 스크립트는 1단계 직후 바로 세팅 | 개발 중 데이터부터 안전하게 보관 |
